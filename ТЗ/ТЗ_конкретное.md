<a id="intro"></a>
## 1. Введение

### 1.1. Назначение документа
Этот документ фиксирует требования к разработке мобильного приложения для трейдинга на бирже и серверной платформы, которая собирает, хранит и предоставляет биржевые данные (котировки, инструменты, операции, аналитика) для мобильных клиентов.

### 1.2. Цели продукта
- Предоставить пользователю удобный и безопасный доступ к информации об инвестиционном счёте и портфеле.
- Дать возможность просматривать доступные инструменты и совершать покупки/продажи (минимальный торговый сценарий).
- Предоставить историю операций и аналитику портфеля.
- Обеспечить масштабируемость (10 000 одновременных пользователей) и безопасность.

### 1.3. Область охвата (Scope)
**Входит:**
- Android приложение (Kotlin + Jetpack Compose).
- Кроссплатформенное приложение (React Native) с аналогичной функциональностью.
- Backend API (Kotlin + Ktor + корутины + JSON serialization).
- Сбор и доставка котировок (Go сервис).
- Linux driver (C) для интеграции с источником данных/биржей (назначение уточняется, см. раздел “Открытые вопросы”).
- Хранилища и инфраструктура данных: Valkey, PostgreSQL, ClickHouse.
- Наблюдаемость: OpenTelemetry.
- Сервис нагрузочного тестирования, имитирующий нагрузку от мобильных приложений.

**Не входит (на текущем этапе, если не будет уточнено):**
- Веб-терминал/веб-приложение.
- Маржинальная торговля, деривативы, сложные ордера (Iceberg, OCO и т.п.).
- Социальные функции/копитрейдинг.
- KYC/AML-процессы (может потребоваться позже; оставить интеграционные точки).

<a id="product-assumptions"></a>
### 1.4. Продуктовые предпосылки
- Продукт реализует **симуляцию торговли** на основе данных котировок **Московской биржи (MOEX ISS)**.
- Торговля происходит **внутри нашей системы** (внешних брокеров/биржевых торговых подключений нет).
- Исполнение ордеров выполняется **на backend**.
- Поддерживается **один тип ордера: Market**.
- Ордер исполняется **полностью (100%)**, если у пользователя достаточно средств/количества.
- Комиссии отсутствуют.
- Поддерживаемые инструменты: **акции, облигации**.
- Валюта расчётов: **RUB**.
- Дробные акции: **не поддерживаются**.
- Мультивалюта: **не поддерживается**.
- У пользователя **один инвестиционный счёт**.
- Есть **виртуальный денежный счёт**, пополняемый через отдельный API.

## 2. Термины и определения
- **Инвестиционный счёт** — набор балансов (денежные средства) + позиции (инструменты).
- **Инструмент** — акция/облигация, доступная к покупке.
- **Позиция** — объём инструмента во владении пользователя.
- **Котировка** — текущая цена/лучшие цены bid/ask, возможно свечи.
- **Операция** — исполненный ордер/пополнение/списание.
- **Портфельная аналитика** — временные ряды стоимости портфеля, структура по классам/инструментам.

---

## 3. Пользовательские роли
- **Пользователь**: просматривает портфель, инструменты, совершает сделки, анализирует результаты.
- **Администратор/Оператор (опционально)**: мониторинг состояния системы, расследование инцидентов, поддержка пользователей.

<a id="architecture"></a>
## 5. Высокоуровневая архитектура

### 5.0. Архитектурный стиль
- Архитектура: **микросервисная**
- Взаимодействие: **event-driven**
- Паттерн: **CQRS** (разделение write/read модели)

### 5.1. Компоненты
- **Mobile Apps** (Android / RN)
- **API Gateway / Backend (Ktor)**:
  - Auth & Sessions
  - Portfolio Service
  - Instruments Service
  - Orders/Trading Service
  - Operations Service
  - Analytics Service
  - Quotes Streaming API (WebSocket)
- **Quotes Collector (Go)**:
  - подключение к источнику котировок (MOEX ISS, HTTP, JSON)
  - нормализация данных
  - публикация в Valkey Pub/Sub / запись в ClickHouse (по политике)
- **Driver (C, Linux)**:
  - низкоуровневое взаимодействие (например, с провайдером/шиной/устройством/каналом) — уточнить
- **Valkey**:
  - кэш справочников/сессий/портфеля (при необходимости)
  - Pub/Sub для котировок/событий
- **PostgreSQL**:
  - пользователи, счета, позиции, ордера, операции, справочники
- **ClickHouse**:
  - котировки/свечи/агрегаты
  - snapshots стоимости портфеля во времени
- **Observability (OTel)**:
  - трассировки запросов, метрики latency/error rate, бизнес-метрики

### 5.2. Потоки данных
1) **Котировки**
- Источник (MOEX ISS, HTTP) → (C driver, если требуется) → Go Collector → Valkey (Pub/Sub) → Backend WS → Mobile
- Дополнительно: запись агрегатов (свечи, OHLCV) в ClickHouse

2) **Портфель/Сделки**
- Mobile → Backend REST → PostgreSQL (ордер/операция)
- События исполнения (если есть) → Backend → обновление позиций → публикация изменений (опционально WS)

<a id="screens"></a>
## 7. Экранные требования (MVP + расширения)

> Примечание: перечисленные экраны обязательны; торговые элементы описаны как минимально необходимые для “покупки инструмента”.

### 7.1. Главный экран (Портфель)
**Цель:** быстро показать состояние счёта и владения.

**Отображает:**
- Текущий инвестиционный счёт:
  - доступные средства
  - текущая стоимость портфеля (equity)
  - прибыль/убыток (P&L) за день и за период (если доступно)
- Список инструментов во владении:
  - тикер/название
  - количество
  - текущая цена
  - P&L по позиции (если доступно)
- Быстрые действия:
  - “Пополнить” (опционально, если есть процесс)
  - “Купить” / “Найти инструмент”

**Acceptance Criteria:**
- При открытии экрана данные загружаются ≤ 2 сек при нормальной сети (цель).
- При ошибке API отображается сообщение и кнопка повторить.

---

### 7.2. Страница “Инструменты” (каталог доступных к покупке)
**Цель:** найти инструмент и перейти к покупке/деталям.

**Функции:**
- Список инструментов с фильтрами:
  - поиск по тикеру/названию
  - категории (акции/облигации)
  - сортировка (по объёму, изменению, популярности — опционально)
- Карточка инструмента:
  - тикер, название
  - текущая цена, изменение % за день
- Переход на экран инструмента

**Acceptance Criteria:**
- Поиск работает по подстроке, задержка ≤ 300 мс локально (при наличии индекса/кэша) либо серверно.

---

### 7.3. Экран инструмента (детали + покупка/продажа)
**Цель:** показать котировки, базовую аналитику и дать купить/продать.

**Отображает:**
- Цена (last), bid/ask
- Мини-график (линия или свечи) за периоды: 1D, 1W, 1M, 1Y (минимум 1D/1W для MVP)
- Ключевые данные (опционально): капитализация, сектор, описание

**Торговый билет (MVP):**
- Тип операции: Купить / Продать
- Тип ордера: Market
- Количество (лоты, целое)
- Проверки:
  - достаточно средств для покупки
  - достаточно количества для продажи
- Подтверждение операции
- Результат: статус “Принят” + отображение в “Операциях”

**Acceptance Criteria:**
- Котировки обновляются в реальном времени при активном экране (WebSocket).
- Создание ордера возвращает идентификатор и статус.

---

### 7.4. Страница “Профиль пользователя”
**Отображает:**
- Основные данные профиля (имя/почта/телефон — по наличию)
- Настройки:
  - безопасность (смена пароля, MFA — если реализовано)
  - уведомления (котировки/исполнения)
  - выход из аккаунта

**Acceptance Criteria:**
- Выход очищает локальные токены и кэш приватных данных.

---

### 7.5. Страница “Операции пользователя”
**Отображает:**
- Список операций/сделок:
  - дата/время
  - инструмент (если применимо)
  - тип (buy/sell/deposit/reset)
  - сумма/количество
  - статус (filled/rejected)
- Фильтры по периоду и типу

**Acceptance Criteria:**
- История подгружается постранично (pagination) или бесконечной лентой.
- При открытии операции доступны детали (при необходимости).

---

### 7.6. Страница “Аналитика портфеля”
**Отображает:**
- График стоимости портфеля во времени (equity curve)
- Структура портфеля:
  - по инструментам (pie/bar)
  - по классам активов (если есть)
- Метрики (обязательные):
  - доходность за период
  - P&L
  - XIRR
  - волатильность
  - max drawdown

**Acceptance Criteria:**
- График строится из агрегированных данных, не блокирует UI.
- Периоды: 1W/1M/1Y (минимум 1M для MVP при наличии данных).

## 8. Требования к API (черновой контракт)

### 8.1. Протоколы
- REST/JSON для большинства запросов.
- WebSocket для стриминга котировок и (опционально) событий портфеля/ордеров.

<a id="api-minimal"></a>
### 8.2. Минимальные группы эндпоинтов
1) **Auth**
- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout`

2) **Profile**
- `GET /me`
- `PATCH /me` (опционально)

3) **Portfolio**
- `GET /accounts`
- `POST /accounts/deposit`
- `POST /accounts/reset`
- `GET /portfolio/summary`
- `GET /portfolio/positions`

4) **Instruments**
- `GET /instruments?query=&type=&limit=&offset=`
- `GET /instruments/{id}`

5) **Quotes**
- `WS /quotes`:
  - subscribe/unsubscribe на список instrumentId
  - сообщения котировок (last, bid, ask, ts)
  - лимит подписок: максимум 50 инструментов на пользователя/соединение
  - при превышении: ошибка `LIMIT_EXCEEDED`
- `GET /candles?instrumentId=&period=&from=&to=` (для графиков)

6) **Orders / Trading**
- `POST /orders` (buy/sell, type=market, qty)
- `GET /orders?status=&from=&to=`
- `GET /orders/{id}`

7) **Operations**
- `GET /operations?from=&to=&type=&limit=&offset=`

8) **Analytics**
- `GET /analytics/portfolio/value?period=`
- `GET /analytics/portfolio/allocation?period=`

<a id="nfr"></a>
## 10. Нефункциональные требования

### 10.1. Производительность и масштабирование
- Система должна выдерживать **10 000 одновременных пользователей**.
- Целевые показатели (ориентиры, фиксируются как SLO):
  - REST API p95 latency: ≤ 300–500 мс на чтение (внутренний DC), ≤ 800 мс на запись ордера (MVP)
  - WebSocket: доставка котировки p95 ≤ 300 мс от момента поступления в backend (без гарантий источника)
- Горизонтальное масштабирование backend (stateless сервисы).
- Контроль нагрузки:
  - rate limiting на пользователя/IP
  - backpressure на WS (ограничение подписок/частоты)

### 10.2. Надёжность
- Целевой аптайм (ориентир): 99.9% для API.
- Graceful degradation:
  - при падении ClickHouse — приложение работает с базовыми данными портфеля (PostgreSQL)
  - при проблемах WS — fallback на периодический polling котировок (умеренный)

### 10.3. Безопасность
- TLS везде (mobile ↔ backend, сервис-сервис).
- Хранение секретов: vault/secret manager (конкретный инструмент выбирается).
- Токены:
  - Access + Refresh, короткий TTL для access
  - ротация refresh
- Защита мобильных клиентов:
  - безопасное хранение токенов (Android Keystore / iOS Keychain; в RN через безопасное хранилище)
- OWASP:
  - ориентироваться на OWASP MASVS (mobile) и OWASP ASVS (backend)
- Аудит действий:
  - логирование входов, торговых операций, критичных изменений профиля
- Изоляция данных:
  - строгая проверка доступа (account_id принадлежит user_id)
- Защита от атак:
  - WAF/anti-bot (по необходимости)
  - CSRF не актуален для pure mobile token auth, но учесть replay-подписи при необходимости

### 10.4. Наблюдаемость (OpenTelemetry)
- Трассировки:
  - запросы REST, операции ордеров, критичные цепочки
- Метрики:
  - latency, error rate, RPS
  - количество активных WS соединений
  - количество подписок на инструмент
  - лаг котировок (collector → backend → клиент)
- Логи:
  - структурированные, корреляция по trace_id

<a id="load-testing"></a>
## 12. Сервис для имитации нагрузки мобильных приложений

### 12.1. Назначение
Сервис генерирует нагрузку, имитирующую поведение мобильных клиентов:
- REST запросы (авторизация, портфель, операции, каталог инструментов)
- WebSocket подключения и подписки на котировки
- (опционально) создание ордеров в тестовом контуре

### 12.2. Требования
- Поддержка сценариев:
  1) “Просмотр портфеля” (login → portfolio/positions → operations)
  2) “Скролл каталога” (instruments list + search)
  3) “Реалтайм котировки” (WS connect → subscribe N инструментов → поддержание соединения)
  4) “Торговля” (create order → check status → operations) — только в тестовой среде
- Распределение сценариев (для нагрузочного профиля):
  - 80%: смотрят портфель/котировки (REST + WS)
  - 15%: листают каталог инструментов (REST + поиск)
  - 5%: торгуют (создают Market-ордера)
- Конфигурация:
  - число виртуальных пользователей (VUs)
  - распределение сценариев (%)
  - длительность теста
  - профиль подписок (сколько инструментов на пользователя)
- Метрики и отчёты:
  - p50/p95/p99 latency по эндпоинтам
  - WS: количество reconnect, лаг сообщений, потери
  - error rate по типам ошибок
- Интеграция в CI/CD:
  - запуск по тегу/пайплайну
  - артефакт отчёта (JSON/HTML)
- Запуск:
  - локально (dev)
  - в Kubernetes/CI (предпочтительно)
- Инструментарий (варианты):
  - k6 / Gatling / Locust (выбрать и зафиксировать; k6 часто удобен под API+WS)

<a id="vtp-pipeline"></a>
### 15.3. Биржевой контур "HTTP -> протокол -> драйвер"

#### 15.3.1. Назначение драйвера C (Linux)
- Драйвер нужен для **получения котировок через виртуальное устройство**, читаемое user-space сервисом.
- Подключение к биржевому протоколу напрямую **не выполняется**.
- Мы **имитируем** биржевой протокол (VTP) и "кормим" им драйвер.

#### 15.3.2. Компоненты
1) **MOEX HTTP Fetcher (Go)**
- опрашивает MOEX ISS по HTTP API;
- преобразует данные в сообщения VTP;
- передаёт сообщения в драйвер (через write/ioctl к char device — целевой вариант).

2) **VTP Driver (C, Linux char device)**
- виртуальное устройство `/dev/vtp_moex0`;
- поддерживает `read()` (и `write()` или `ioctl()` - в зависимости от выбранного способа инжекта данных);
- ring buffer + счётчики потерь.

3) **Quotes Reader/Collector (Go/Kotlin)**
- читает `/dev/vtp_moex0`;
- публикует котировки в **Valkey Pub/Sub** для backend WS;
- пишет raw ticks в ClickHouse (TTL см. раздел 9.2).

#### 15.3.3. Предложение "виртуального протокола" (VTP-1)
Протокол предлагается как бинарный фрейм для эффективности и простоты драйвера.

##### Формат сообщения (TickFrame v1)
- Magic: 2 bytes (`0x4D 0x58` = "MX")
- Version: 1 byte (`0x01`)
- MsgType: 1 byte (`0x01` = tick)
- SeqNo: 8 bytes (uint64, монотонный)
- TsUnixMs: 8 bytes (uint64)
- InstrumentId: 8 bytes (uint64) *(или хэш/код, если справочник фиксирован)*
- Price: 8 bytes (int64) цена в копейках
- Qty: 8 bytes (uint64) количество в лотах
- Side: 1 byte (`0` = unknown, `1` = buy, `2` = sell)
- CRC32: 4 bytes (контроль целостности фрейма)

##### Требования к драйверу
- Экспортировать `/dev/vtp_moex0`
- Поддержать `read()` для пользовательских процессов:
  - блокирующее чтение (по умолчанию)
  - неблокирующее (если выставлен O_NONBLOCK)
- Внутренний ring buffer:
  - ограничение по памяти
  - стратегия при переполнении: drop oldest + счётчик потерь

##### Требования к Fetcher Service
- Периодически запрашивает MOEX и преобразует события в TickFrame v1.
- Записывает фреймы в источник, доступный драйверу:
  - вариант A (предпочтительно): запись в `/dev/vtp_moex0` через ioctl/write, если драйвер поддержит
  - вариант B: shared memory / netlink (сложнее)
- Для MVP можно упростить:
  - fetcher пишет в user-space buffer, а драйвер "подбирает" данные через shared file/socket (если допустимо), но целевое решение - настоящий char device.

> Примечание: окончательный способ "как данные попадают в драйвер" нужно зафиксировать при проектировании драйвера (write/ioctl vs вспомогательный канал).

<a id="moex-integration"></a>
### 15.5. Интеграция с MOEX ISS: какие данные забираем и как "слушаем"

#### 15.5.1. Принцип
MOEX ISS — HTTP API. "Слушать биржу" в рамках ISS реализуется как **частый polling + инкрементальные обновления через SEQNUM/dataversion**.
- В ответах могут приходить блоки `dataversion` и служебное поле `SEQNUM`, которые следует использовать для получения только изменений, а не полной выгрузки.
- Нужно отслеживать изменения версии (dataversion): при смене торгового дня SEQNUM может "сброситься/уменьшиться", это надо корректно обработать.
- Структура ответа не фиксирована по позициям; парсинг делаем по `id` блоков/именам полей, не "по месту".

#### 15.5.2. Что именно забираем с биржи (минимум, но достаточно для продукта)

##### 15.5.2.1. Справочники (Reference Data) — 1 раз в сутки
Цель: иметь каталог инструментов (акции/облигации), режимы торгов и связки BOARDID+SECID.
- Дерево параметров `engine -> market -> board/boardgroup -> security` строим динамически, как рекомендует ISS.
- Для ускорения можно брать "общий индексный ответ" (пример: `iss/index`, где видны сущности рынков/режимов/коллекций).
- По справочникам поддерживаем только рынки/инструменты, соответствующие **акциям и облигациям**.

Сохраняем в PostgreSQL:
- instrument: SECID, shortname, fullname, type (share/bond), lot size, currency=RUB
- торговые связки: BOARDID, MARKET, ENGINE, SECID (ключ — BOARDID+SECID)

##### 15.5.2.2. "Raw ticks" (сырьё) — основной поток
Цель: иметь максимально близкое к реальности "тик-поток" для построения свечей/графиков и для realtime UI.
- **Основной источник raw ticks = реестр сделок (`/trades`)** (в ISS указано как тип данных "сделки (анонимно)" и объект `/trades`).
- Пишем в ClickHouse raw ticks (TTL см. раздел 9.2).

RawTick (нормализованная схема):
- ts_exchange (или ts_iss), secid, boardid
- price (в копейках), qty (лоты), side (если есть)
- seqnum, dataversion_version (для дедупликации/инкремента)
- raw_payload (опционально: исходный JSON для диагностики)

Важно по ценам:
- учитывать `DECIMALS/PRECISION` из ответов ISS для корректного округления/интерпретации цены.

##### 15.5.2.3. Котировки bid/ask (опционально, "улучшение")
- Источник: `orderbook` (в ISS объект `/orderbook`).
- Используем **только для UI** (показывать bid/ask) и для fallback "средней" цены, если LAST отсутствует.
- Не делаем это обязательным для MVP, потому что **доступ к рыночным данным без аутентификации может быть с задержкой, а `orderbook` доступен только авторизованным**.

---

#### 15.5.3. Правило цены исполнения (симуляция)
- Market-ордер исполняется по `LAST` из нашего потока (raw ticks / вычисленный last).
- Если `LAST` отсутствует: используем "среднее":
  - при наличии bid и ask: `(bid + ask)/2`,
  - иначе: последняя известная цена инструмента (из нашего кэша последней цены).

---

#### 15.5.4. Как делаем polling (чтобы это было реально "как поток")

##### 15.5.4.1. Инкрементальная схема
- Для каждого "источника таблицы" (например, trades по рынку/режиму) храним `(dataversion, last_seqnum)`.
- Следующий запрос выполняем с параметром `seqnum=last_seqnum`, чтобы получить только обновления.
- При смене `dataversion` (новый торговый день) сбрасываем курсор и начинаем заново с актуального SEQNUM.
- Если ответ порционный - используем `start/limit` и цикл до пустого блока данных.

##### 15.5.4.2. Частота polling
- Trades polling: 0.5–2 секунды (настраиваемо).
- Orderbook polling (если включим): только для "горячих" инструментов (те, что сейчас подписаны пользователями), например раз в 1–2 секунды.

##### 15.5.4.3. Оптимизация ответов ISS (обязательная)
- Используем системные параметры, чтобы не тянуть метаданные и лишние блоки:
  - `iss.meta=off`, `iss.json=compact`
  - `iss.only=...` (только нужные блоки)
  - `<block>.columns=...` (только нужные поля)
- Включаем HTTP-сжатие (gzip), ISS это поддерживает.
- При 500–504 делаем retry с backoff.

---

#### 15.5.5. Доступность данных (подписка/аутентификация)
- Для режима "без подписки" возможно получение рыночных данных с задержкой; для `orderbook` требуется авторизация.
- В продукте фиксируем:
  - MVP допускает delayed-данные (если нет подписки),
  - продовый режим "near real-time" требует подписки и включенной аутентификации.
